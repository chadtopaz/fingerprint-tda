# Fingerprint verification via topological data analysis — replication code

Replication code for:

> Topaz, C. M., Arcila-Maya, N., Munch, E., Stanley, Z., and Ziegelmeier, L. *Topological summaries of fingerprint ridge patterns carry identity information.* Manuscript in submission, 2026.

**Repository:** <https://github.com/chadtopaz/fingerprint-tda>

This repository contains the analysis and figure-generation code. It does **not** include the fingerprint data (a third-party benchmark — see [Data](#data)) or any generated outputs (they are regenerated by running the code).

## What this code does

Starting from enhanced binary ridge/valley fingerprint images, the pipeline computes two Euclidean distance transforms, applies sublevel-set persistent homology (via GUDHI) to each, and vectorizes the resulting diagrams as Betti curves and persistence images. It then evaluates seven verification methods — two geometry baselines, two global topological baselines, a trained logistic-regression scorer (TopoLR), a local optimal-transport matcher (LPOT), and their fusion — on FVC2000 DB1 under identity-disjoint 5-fold cross-validation with a strict no-leakage protocol. See the paper for full detail.

## Data

This study uses **FVC2000 DB1** — the full set of 100 fingers x 8 impressions = 800 images (the "DB1_A" evaluation set) — a third-party benchmark we do not redistribute. Cite it as Maio et al. (2002). The code reads *enhanced binary* ridge/valley images derived from it, so reproducing the pipeline takes three steps:

1. **Obtain FVC2000 DB1.** The full 100-finger DB1 is distributed on the DVD accompanying the *Handbook of Fingerprint Recognition* (Maltoni, Maio, Jain, and Feng; Springer, 3rd ed., 2022). Competition details and a small free subset (DB1_B, 10 fingers) are at the FVC2000 site, <http://bias.csr.unibo.it/fvc2000/>.
2. **Enhance to binary.** Apply Hong-Wan-Jain enhancement, as implemented in the Fingerprint-Enhancement-Python library (Utkarsh Deshmukh, <https://github.com/Utkarsh-Deshmukh/Fingerprint-Enhancement-Python>) with default parameters, to obtain binary ridge/valley images.
3. **Place and name the files.** Save the 800 binary PNGs in `Data/FVC2000_binary/` named `<finger>_<impression>_bin.png` (`1_1_bin.png` ... `100_8_bin.png`). The loader reads every `*.png` in that directory and infers the finger identity from the filename prefix; ridge pixels are taken to be the minority class, and the loader inverts the labeling if needed.

See `Data/README.md` for more. Exact bit-for-bit reproduction of the binary images depends on the enhancement library version; given a fixed set of binary images, the topological pipeline below is exactly reproducible.

## Dependencies

- **R** (>= 4.2 recommended).
- CRAN: `tidyverse`, `TDA`, `TDAvec`, `digest`, `future`, `future.apply`, `matrixStats`, `pROC`, `progressr`.
- Bioconductor: `EBImage` (image I/O and distance transforms).
- Persistent homology uses **GUDHI**, which is bundled with the `TDA` package — no separate install.

Install everything with:

```r
source("DEPENDENCIES.R")
```

For exact reproducibility, record your environment with `sessionInfo()`.

## How to reproduce

Open `fingerprint-tda.Rproj` in RStudio (this sets the working directory to the repository root), or `setwd()` to the repository root, then:

```r
# 1. Main analysis. On the first run this builds the feature-bank cache
#    (slow: minutes to tens of minutes), then scores all seven methods under
#    5-fold CV, writes every result CSV to Outputs/, and writes the ROC figure.
#    Later runs reuse the cache and are fast.
source("Code/fingerprintTDA.R")

# 2. Figures: pipeline, local patches, score distributions.
source("Code/make_figures.R")

# 3. (optional) Downsampling validation and the PCA-matched control.
source("Code/validate_downsampling.R")
source("Code/pca_match_analysis.R")
```

### Reproducibility

All random seeds and iteration counts are fixed (`CONFIG$cv_seed`, the TopoLR SGD seed, per-image anchor seeds derived from filenames, and fixed Sinkhorn iterations with no early stopping). Given the same binary images, the pipeline produces identical numerical results across runs. Parallelism (the `future` package) affects wall-clock runtime only, not results.
